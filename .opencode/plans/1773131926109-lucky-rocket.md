# ITM Middle-Override Bug Implementation Plan

> **For agentic workers:** REQUIRED: Use `superpowers:subagent-driven-development` (if subagents are available) or `superpowers:executing-plans` to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Fix `ItmDetector` so it detects ITMs for the analyzed plugin even when that plugin is not the winning override in a full load-order cache.

**Architecture:** Keep the current `ResolveAllSimpleContexts(...)` approach, but stop assuming the analyzed plugin is at index `0`. Instead, locate the analyzed plugin's context in the resolved override chain and compare it to the immediately previous version it overrides. This plan intentionally broadens `ItmDetector` to work with normal full load-order caches. Update docs/comments so the cache contract matches actual behavior, and add regression tests for 3-mod load orders.

**Tech Stack:** C# 13, .NET 10, Mutagen.Bethesda 0.53.1, xUnit, FluentAssertions

## Implementation Plan

### Task 1: Reproduce the bug with a failing regression test

**Files:**
- Modify: `QueryPlugins.Tests/Detectors/ItmDetectorTests.cs`

- [ ] Add a 3-mod helper shape such as `Master.esm -> Plugin.esp -> Later.esp` that can build the analyzed plugin as a middle override in the load order.
- [ ] Add a regression test where `Plugin.esp` is identical to `Master.esm`, `Later.esp` overrides the same record with a real change, and analysis still reports an ITM for `Plugin.esp`.
- [ ] Add a companion test where `Plugin.esp` changes the record relative to `Master.esm`, `Later.esp` also overrides it, and analysis reports no ITM.
- [ ] Keep existing 2-mod tests intact so the current winner-path behavior remains covered.

### Task 2: Fix `ItmDetector` context selection

**Files:**
- Modify: `QueryPlugins/Detectors/ItmDetector.cs`

- [ ] Keep the current early skips for new records and deleted records.
- [ ] After `ResolveAllSimpleContexts(...)`, find the context whose `ModKey` matches `plugin.ModKey` instead of assuming the analyzed plugin is at index `0`.
- [ ] If the plugin context is missing, skip gracefully as today.
- [ ] If the plugin context is the last item in the resolved chain, skip because there is no lower-priority version to compare against.
- [ ] Compare the analyzed plugin's record to the next item in the winner-first chain, because that next item is the version it directly overrides.
- [ ] Update the inline comments to describe Mutagen's ordering and the new comparison rule clearly.

### Task 3: Align public docs with the fixed behavior

**Files:**
- Modify: `QueryPlugins/Detectors/IItmDetector.cs`
- Modify: `QueryPlugins/IPluginQueryService.cs`

- [ ] Replace the current "plugin must be the winning override" wording.
- [ ] Document that the analyzed plugin must be present in the provided load-order cache.
- [ ] Document that ITM comparison is against the immediately lower-priority context for that record, not necessarily the overall winner or origin.
- [ ] Check whether any related implementation comments in `QueryPlugins/PluginQueryService.cs` repeat the old assumption; update only if needed.
