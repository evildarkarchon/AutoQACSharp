This is harder to test directly because the retry is for real file contention. Two approaches:
- Option A: Verify that the method succeeds when the file is accessible (happy path already covered). Trust the retry logic through code review.
- Option B: Lock the file with a `FileStream` that prevents reading, verify the first attempt fails, release after a delay, verify eventual success.

Implement Option B if feasible:
```csharp
[Fact]
public async Task ReadLogContentAsync_RetriesOnIOException_EventuallySucceeds()
{
    var logPath = Path.Combine(_testRoot, "SSEEdit_log.txt");
    File.WriteAllText(logPath, "log content\r\n");

    // Lock the file exclusively
    using var lockStream = new FileStream(logPath, FileMode.Open, FileAccess.ReadWrite, FileShare.None);

    // Start the read on a background task
    var readTask = Task.Run(async () =>
        await _sut.ReadLogContentAsync(_testRoot, GameType.SkyrimSe, 0, 0));

    // Release after 150ms (first retry at 100ms will fail, second at 200ms should succeed)
    await Task.Delay(150);
    lockStream.Dispose();

    var result = await readTask;
    result.LogLines.Should().NotBeEmpty();
}
```

If this test is flaky due to timing, add a comment noting it tests retry behavior and may need adjustment. Alternatively, skip this test with `[Fact(Skip = "Timing-sensitive retry test")]` and note it was verified manually.
